## Part 1. 기본 용어 사전 (왕초보 필독)

이 프로젝트에 등장하는 주요 개념을 비유로 정리합니다.

### 1-1. 이 프로젝트는 뭐하는 곳인가?

**한 줄**: 음성 대본(mp3 + 자막 SRT) → 자동으로 유튜브 영상(mp4) 생성.

**공장 비유**:
```
재료                    가공                          완성품
───────────           ──────────                  ───────────
mp3 오디오     →       Claude AI 가 씬 배치  →    mp4 영상
SRT 자막              (디자인 + 모션)
```

### 1-2. "씬(scene)" 이 뭐냐?

영상의 한 장면. 대략 5~15초 길이. 화면에 **무엇이 보이고 어떻게 움직이는가** 를 정의합니다.

- 예) 씬 0: "안녕하세요 바이브랩스입니다" — VIBELABS 워드마크 + 요일 인디케이터
- 예) 씬 3: "뉴스 1 — 토큰이 바닥났다" — 큰 빨간 01 + 타자기 헤드라인

### 1-3. Remotion 이 뭐냐?

**React 로 영상을 만드는 프레임워크**. 영상의 각 프레임을 React 컴포넌트로 그립니다.

```
일반 웹사이트 React                Remotion
────────────────                   ─────────────
<div>Hello</div>  →  브라우저      <div>Hello</div>  →  mp4 프레임
(한 시점의 화면)                   (시간 축의 연속 프레임)
```

각 프레임 번호 `frame` 변수가 주어지면, 그 시점의 화면을 JSX 로 그립니다. `interpolate(frame, [0, 30], [0, 1])` 이라고 쓰면 프레임 0~30 동안 값이 0→1 로 변한다는 뜻. 이걸로 애니메이션을 만듭니다.

### 1-4. Composition / Sequence / TransitionSeries ?

Remotion 의 영상 구조.

- **Composition**: 영상 하나 전체 (예: 12분짜리 뉴스 영상).
- **Sequence**: Composition 안의 구간 (예: 씬 하나).
- **TransitionSeries**: 여러 Sequence 를 연결하면서 사이에 전환(crossfade/wipe 등)을 넣는 컨테이너.

```
Composition (영상 전체)
└── TransitionSeries
    ├── Sequence 0 (씬 0, 9초)
    ├── Transition (crossfade, 20 프레임)
    ├── Sequence 1 (씬 1, 12초)
    ├── Transition (slide-right, 20 프레임)
    ├── Sequence 2 (씬 2, 9초)
    ...
```

### 1-5. DSL 이 뭐냐? (이게 핵심!)

**DSL = Domain Specific Language**. 특정 도메인 전용의 작은 언어.

이 프로젝트의 DSL 은 **JSON 트리** 였습니다:
```json
{
  "type": "SceneRoot",
  "children": [
    { "type": "Kicker", "data": { "text": "주간 체감량" } },
    { "type": "Split", "children": [
      { "type": "ImpactStat", "data": { "value": 55, "label": "월요일" } },
      { "type": "CompareBars", "data": { "items": [...] } }
    ]}
  ]
}
```

이걸 `stack_root` 라고 부릅니다. 74 종류의 노드 타입 (`Kicker`, `ImpactStat`, `CompareBars`, `SvgAsset` 등) 중에서 골라 **레고 블록처럼 조합**해서 씬을 만듭니다.

**레고 블록 비유**:
- 블록 74 종류 → 조립 매뉴얼(`pattern_ref`) 에 따라 → 집/차/배 모양 완성.
- 장점: 규칙이 명확. 검증 쉬움.
- 단점: 블록 카탈로그에 없는 모양은 못 만듦.

### 1-6. TSX 가 뭐냐?

**TypeScript + JSX**. React 컴포넌트 파일 확장자.

DSL(JSON) 과 달리 TSX 는 **자유 그림판**:
```tsx
export const Scene03: React.FC = ({ frame }) => {
  return (
    <AbsoluteFill>
      <div style={{ position: "absolute", left: 100, top: 200,
                    fontSize: 96, color: "red" }}>
        NEWS 01
      </div>
      {/* 원하는 모양 자유롭게 그릴 수 있음 */}
    </AbsoluteFill>
  );
};
```

**자유 그림판 비유**:
- 크레파스·연필·펜 자유 선택 → 원하는 대로 그림 → 유일무이한 그림.
- 장점: 표현 자유도 무한.
- 단점: 규칙/검증이 어려움 (아무렇게나 그릴 수 있음).

### 1-7. scene-grammar 가 뭐냐?

**이 프로젝트의 "영상 문법서"**. `.claude/rules/scene-grammar.md` 에 있습니다.

v1.0 (2026-04-17) 부터 v1.4 (2026-04-19) 까지 계속 발전. Claude AI 가 씬을 만들 때 따라야 할 **규칙 + 검증기 + 원칙** 을 담음.

### 1-8. validator / HARD GATE / hook 이 뭐냐?

**validator**: "이 씬이 규칙 지켰나?" 자동 검사 스크립트. 예) `validate-node-count.js` 는 "씬에 의미 노드가 5~9개 사이인가?" 검사.

**HARD GATE**: "통과 못 하면 렌더 금지" 수준의 엄격한 validator. `exit 1` 로 파이프라인 중단.

**hook**: 특정 시점에 자동 실행되는 스크립트. 예) pre-commit hook 은 `git commit` 직전에 돌아서, 실패하면 커밋 막음.

### 1-9. reference/SC *.png 이 뭐냐?

**영상의 시각적 DNA 단일 원천**. 61장의 PNG 이미지가 `reference/` 폴더에 있음. 디자이너가 수작업으로 만든 레이아웃 샘플. Claude 는 이걸 보고 스타일을 내재화.

### 1-10. Claude / Opus / 1M context ?

**Claude**: Anthropic 이 만든 AI 모델. 이 프로젝트의 씬 설계를 담당.
**Opus**: Claude 의 최고급 라인 (가장 똑똑하지만 비싼).
**1M context**: 한 번에 들고 있을 수 있는 "기억" 용량. 1M token ≈ 75만 단어 ≈ 책 6~7권 분량.

200K context Claude 면 "책 1~2권" 만 들고 있음. 1M 은 "reference 이미지 61장 + 전체 프로젝트 코드 + 대본 + 예시 20개" 를 동시에 들고 있을 수 있음.

## Part 2. 어제까지의 세계 — 무엇이 안 되었나

### 2-1. 증상

한 마디로: **"AI 가 만든 영상이 단조롭다"**.

구체적으로:
- **쌍둥이 구도**: 다른 씬인데 똑같이 생김. 감사에서 `scene-22 ↔ scene-68` 이 둘 다 "링 차트 3개 나란히" (trio-RingChartx3 쌍둥이) 판정.
- **near-empty 씬**: 화면에 배지 1개만 덩그러니 있는 빈 느낌. 79프레임 중 **35.4%** 해당.
- **dead-space**: 하단 50% 가 비어있는 상단 편향 구도. **82%** 씬이 해당.
- **자막과 화면 텍스트 중복**: 자막에 "다음에 또 만나요" → 화면 큰 글씨도 "다음에 또 만나요" (똑같음).
- **사각형 나열**: 카드 + 카드 + 카드. 슬라이드쇼 같음.
- **엔딩 블랙**: 영상 마지막 5 프레임이 완전 검정 (scene-77 버그).

### 2-2. 왜 문제인가?

CLAUDE.md 의 디자인 철학을 보면 명확합니다:

> **이 프로젝트의 적은 "무난함"이다.**
> AI 가 만드는 영상은 기본적으로 단조롭다. 통계적 평균으로 수렴하기 때문이다.
> 우리의 목표는 "AI 가 만든 것 같지 않은 영상"이다.

AI 가 "평균" 으로 수렴하면 → 시청자가 "또 AI 영상이네" 느낌 → 차별화 실패.

### 2-3. 대응 노력 (실패한 패턴)

R1 (2026-04-11) 부터 R11 (2026-04-19) 까지 약 8일간 **"규칙 추가 사이클"** 을 반복:

```
문제 발견 → validator 하나 추가 → 새 문제 발견 → validator 또 추가 → ...
```

결과물:
- **21 종 HARD GATE** (v1.0)
- **27 종 → 33 종** (v1.1, 2026-04-18, 42 씬 authoring 후)
- **v1.2 픽셀 게이트** (2026-04-19, 79 프레임 감사 후 5종 추가)

각 validator 는 나름 잘 작동했습니다. `validate-node-count.js` 는 ZERO-TOLERANCE 로 작동. `validate-layout-diversity.js` 도 shape-hash 반복 차단.

**하지만** 감사 결과는 여전히 35.4% near-empty, 82% dead-space. 왜?

### 2-4. 근본 원인 — DSL 의 표현력 한계

핵심 깨달음:

> **Schema 를 통과하는 것과 픽셀이 통과하는 것은 다르다.**

74 노드를 아무리 잘 조합해도, **조합의 총 경우의 수가 유한**합니다. 그리고 Claude 는 어차피 통계적 분포에서 "자주 쓰이는 조합" 을 골라냅니다. 결국:

```
Pattern P04 ("링 차트 + 불릿 리스트") 를 realize →
  RingChart + BulletList 가 비슷한 위치에 비슷한 크기로 배치됨.
```

이게 다른 씬에서 반복되면 `trio 쌍둥이` 증상이 나옵니다.

Validator 를 늘려도 해결이 안 되는 이유:
- validator 는 "이미 만들어진 조합" 을 검사하지, "새로운 조합 가능성" 을 창출하지 못함.
- 검사 통과하는 조합만 골라 쓰면 → 표현 공간이 **더 좁아짐**.
- 좁아진 공간에서 다양성을 찾으려니 → 블록의 색깔/크기만 바꾸는 수준에 머무름.

### 2-5. 비유: 레고 블록 공장

74 블록으로 집을 짓는다고 상상해보세요. 규칙은:
- 빨간 블록 5개 이상 쓰지 마라
- 높이 10층 이하
- 창문 블록은 홀수 개만
- 창문 위치 대칭 금지
- ...

규칙이 늘수록 **지을 수 있는 집 모양이 줄어듭니다**. 어느 순간부터는 규칙 전부 만족하는 집이 몇 개밖에 없고, 결국 비슷한 집들만 반복해서 짓게 됩니다.

**그 상태가 바로 어제까지의 상황이었습니다.**

### 가설 A: hyperframes 오픈소스 덕분

#### A-1. hyperframes 가 뭐냐?

[Hyperframes](https://github.com/...) 는 HTML + GSAP 기반의 영상 제작 오픈소스. Remotion 과는 다른 스택이지만, **"프로그래밍으로 영상을 만든다"** 는 같은 계보.

2026-04-17 경 hyperframes v1.0 의 "장면 조립 수칙 11장" 을 우리 프로젝트에 흡수:
- Phase A/B (레이아웃 → 모션 분리)
- 정량 가드레일 (첫 enterAt ≥ 3f, 씬당 이징 ≥ 3종 등)
- DESIGN.md 존재 요구
- 자막-시각 중복 금지
- ...

#### A-2. 무엇을 바꿨나

직접 기여는 2개:
1. **scene-grammar.md 의 철학적 골격** — v1.0 상당 부분이 hyperframes 원문 이식.
2. **"픽셀을 봐라"** 마인드셋 — Schema 통과 ≠ 픽셀 통과 선언.

#### A-3. 증거

✅ **지지 증거**:
- `.claude/rules/scene-grammar.md` 헤더에 "출처: Hyperframes v1.0 장면 조립 수칙" 명시.
- Phase A/B 강제, preview 스냅샷, Layout before Animation — 전부 hyperframes 원문에서 온 것.

❌ **반증 증거**:
- hyperframes 흡수한 2026-04-17 이후에도 2일간 (04-17~04-19) 문제 지속.
- 흡수했지만 여전히 DSL 프레임워크 안에서만 적용 → 한계 극복 안 됨.
- hyperframes 는 HTML/GSAP, 우리는 Remotion React — 직접 코드 이식 불가.

#### A-4. 기여도 평가: ~20%

**"방향을 알려줬지만, 이동을 시키지는 못했다"**

hyperframes 는 "DSL 에 갇히지 마라" 라는 **철학적 좌표**를 제공했습니다. 하지만 실제로 DSL 을 버리는 결단 자체는 하네스 원칙 A 의 몫이었습니다.

hyperframes 가 없었다면? 아마 "DSL 을 더 많은 validator 로 보강" 하는 쪽으로 갔을 것. hyperframes 의 프레이밍이 "이미 DSL 은 문제다" 라는 인식 전제를 세워줬습니다.

### 가설 B: Claude Opus 4.7 (1M context) 디자인 능력

#### B-1. 1M context 가 왜 중요한가

일반 Claude: 200K context (책 1~2권 분량).
Opus 4.7: **1M context** (책 6~7권 분량).

씬 하나 설계할 때 제가 실제로 들고 있는 정보:
- `scenes-v2.json` 전체 (78 씬의 narration + visual_plan) — ~30K tokens
- `scene-plan.json` — ~20K
- reference/SC 1.png ~ SC 30.png 이미지 Read — ~100K (이미지는 비쌈)
- 기존 TSX 예시 (scene-03/22/26/32/54/77) — ~15K
- `scene-grammar.md` + SKILL.md 전체 — ~40K
- `CLAUDE.md`, `design-system.md` — ~20K
- 노드 타입 정의, theme.ts, StackRenderer.tsx — ~30K
- 대화 이력 — ~100K

**합계**: 약 350K tokens. 200K context 에서는 **절대 불가능**.

200K 였다면: 씬 하나 작성할 때마다 reference 를 골라 Read, 그리고 이전 씬과의 일관성은 기억 밖으로 나감. 결과적으로 씬들이 따로 놀거나, reference 도 일부만 내재화.

#### B-2. 실제 설계 과정

제가 오늘 12 씬을 만들 때 한 일:

```
씬 하나당:
1. narration 읽기 ("100달러 / 8시간 → 1시간")
2. 감정 추출 ("Before/After 극단 대비, 분노")
3. reference 중 톤 맞는 것 상기 (SC 5: 강한 좌우 바 비교)
4. JSX 설계:
   - "왼쪽 mint 8시계, 오른쪽 빨간 1시계가 쪼그라듦"
   - "중간에 gradient arrow + -87% 빨강"
   - "상단 100달러 배지"
5. 코드 작성 (spring, interpolate, Easing 조합)
6. render-single-scene.sh 로 PNG → Read 로 검수
7. 다음 씬으로
```

**이게 200K 에서 됐을까?** 단연코 No. reference 이미지는 매번 다시 Read 해야 하고, 이전 씬과의 구도 비교는 메모리 밖으로 나감.

1M 에서는 **전체 영상 톤 + 모든 reference + 모든 예시 TSX 가 동시에 머릿속**에 있어서 "이 씬은 앞 10 씬과 다르게" 를 자연스럽게 판단.

#### B-3. 증거

✅ **지지 증거**:
- 오늘 작성한 12 씬이 서로 다른 디자인 언어를 가짐 (타자기/영수증/시계/게이지/히트맵...).
- 각 씬 설계 시 reference 여러 장을 동시에 참조한 흔적 (응답에 SC 1/5/12 동시 언급).
- trio 쌍둥이 0건 — 전체 컨텍스트 내재화 없이는 어려운 수준.

❌ **반증 증거**:
- 1M context 는 **2026-04 이전부터 쓸 수 있었음**. 그런데 어제까지 이 문제가 있었음.
- 즉, 1M 만으로는 충분하지 않았음 — **DSL 이라는 표현 제약이 더 큰 병목**이었음.

#### B-4. 기여도 평가: ~30%

**"능력은 있었지만, 발휘될 무대가 없었다"**

Claude Opus 4.7 은 어제도 오늘도 같은 모델입니다. 변한 건 **무대**였습니다. DSL 무대에서는 이 능력이 "74 블록 조합 최적화" 에만 쓰였습니다. TSX 무대에서는 **자유 JSX 설계** 로 쓰입니다.

비유하자면:
- 어제: "당신은 74가지 레고 블록으로 가장 창의적인 집을 지으세요."
- 오늘: "당신은 연필과 종이로 자유롭게 집을 그리세요."

같은 디자이너여도, 후자에서 더 창의적인 결과물이 나옵니다. 전자는 블록 개수의 제약을 받기 때문.

### 가설 C: 하네스 강제력 원칙

#### C-1. 하네스란?

**Harness** = 마구(馬具). 말을 다루는 도구. 여기서는 "AI 의 작업 방향을 규율하는 구조" 를 뜻함.

이 프로젝트의 하네스는 `.claude/rules/scene-grammar.md` + `postprocess.sh` + hooks + validators + SKILL.md 의 종합.

#### C-2. scene-grammar v1.4 의 3개 원칙

**원칙 A — 기본값 전환**
> TSX 가 기본값이다. DSL 은 3조건 동시 만족 시만 허용되는 좁은 예외.

3조건:
1. `data_only`: narration 이 순수 데이터 전달 의도.
2. `pattern_unique`: 해당 pattern 이 프로젝트 내 다른 씬에서 미사용.
3. `no_emotion`: 감정 비트(승부/대비/전환) 없음.

**원칙 B — adapter 패턴 (자산 보존)**
> 74 노드는 `src/remotion/custom/_dsl.tsx` adapter 로 TSX 내부에서 재사용 가능.

코드:
```tsx
import { D } from "./_dsl";

<D type="ImpactStat" data={{ value: "31", suffix: "B" }}
   frame={frame} durationFrames={durationFrames} />
```

이렇게 하면 TSX 자유 JSX 안에서도 `<D type="..." />` 로 DSL 노드 부분 재사용 가능. 74 노드 자산이 그대로 살아남음.

**원칙 C — AST validator 체인**
> 자유 JSX 로 만든 결과물의 **구조만** 검증.

3종 validator:
- `⓪-tsx1 structural-signature`: JSX tag 구조 signature cluster ≤ 2 (쌍둥이 차단)
- `⓪-tsx2 video-narration-match`: `<OffthreadVideo src>` 가 narration 키워드와 매칭
- `⓪-tsx3 text-dedup`: JSX 문자열 ↔ SRT 자막 Levenshtein similarity ≤ 0.6

+ pre-commit hook 이 DSL rationale 미충족 시 commit 차단.

#### C-3. 실제 효과

오늘 실제로 일어난 강제 순간:

**순간 1 — 비디오 키워드 매칭 실패**:
scene-00 에 `futuristic-technology-abstract.mp4` 를 썼는데, narration 에 "미래/추상" 키워드가 없음. `⓪-tsx2` 가 FAIL 하자 → `AI-technology.mp4` 로 교체 (narration 에 "AI" 키워드 있음).

**순간 2 — 자막 복제 감지**:
scene-9 JSX 에 "원인이 뭘까요?" 가 있었는데 SRT 자막에 동일 문장 존재. `⓪-tsx3` Levenshtein similarity 1.000 → FAIL. "정체는 무엇?" 으로 교체.

**순간 3 — DSL rationale 강제**:
기존 22 DSL 씬은 전부 `_dsl_rationale_review_required: true` 플래그 설정. `⓪-dsl validate-dsl-rationale` 가 FAIL 로 잡아냄 → "TSX 전환 후보" 메시지 출력.

이 세 가지 FAIL 이 없었다면 저는 해당 문제들을 간과했을 것입니다.

#### C-4. 증거

✅ **지지 증거**:
- 원칙 A 가 없었다면 저는 오늘도 DSL 로 작업했을 것 (기본값이 DSL 이었으니).
- 원칙 C validator 덕에 scene-9 자막 중복 같은 미세한 실수가 즉시 감지됨.
- pre-commit hook 덕에 DSL rationale 없는 커밋이 아예 차단됨.

❌ **반증 증거**:
- 어제까지도 HARD GATE 33종이 있었는데 문제 지속 — **규칙의 양이 아니라 종류가 중요**.
- 구체적으로, 어제까지의 validator 는 "DSL 내부 조합" 검증. 오늘 validator 는 "TSX 결과물 구조" 검증. 범주가 다름.

#### C-5. 기여도 평가: ~50%

**"표현 층을 해방시키고 결과물 구조만 검증하는 구조"**

원칙 A (표현 층 해방) + 원칙 C (결과물 구조 검증) 의 조합이 어제와 오늘을 가르는 **결정적** 차이입니다. hyperframes 도 Claude 능력도 이 조합 없이는 발휘될 수 없었습니다.

**특히 원칙 A 의 공식 선언이 중요**했습니다. "실험적으로 TSX 도 된다" 정도였다면 저는 계속 DSL 을 썼을 것. "TSX **가** 기본값이다" 는 단호한 선언이 제 선택을 결정했습니다.

### 5-4. 만약 하나가 빠졌다면?

**하네스 없이 Claude + hyperframes 만?**

- ❌ 돌파 못 함.

**Claude 1M 없이 하네스 + hyperframes?**

- 12 씬을 하루에 완성 불가.
- ⚠️ 부분 돌파.

**hyperframes 없이 하네스 + Claude?**
- "픽셀을 봐라" 프레이밍 부재 → 여전히 schema 중심 사고.
- Phase A/B 분리 아이디어 부재 → 정적 검증 없이 모션부터 설계.
- ⚠️ 부분 돌파 (그래도 가능).

## Part 6. 이론적 토대

### 6-1. Karpathy 의 bitter lesson

Andrej Karpathy 가 자주 인용하는 "AI 의 쓴 교훈":

> "더 많은 규칙으로 AI 를 조종하는 건 단기 효과. 장기적으로 이기는 건 **제약을 덜 주고 더 큰 모델/더 많은 데이터** 를 주는 접근."

이 프로젝트 맥락에서:
- **단기 (어제까지)**: DSL + 33 validator 로 품질 통제 시도.
- **장기 (오늘)**: DSL 제거 + TSX 자유 + 소수 AST validator + Claude 능력 신뢰.

AI 가 "74 블록 조합" 수준에서 "자유 JSX 창작" 수준으로 해방되자, **같은 Claude** 가 훨씬 좋은 결과를 만들었습니다. 이게 bitter lesson 의 축소판.

### 6-2. 제약 이론 — 규칙과 창의성의 반비례

**표현 공간 (expression space)** 의 크기가 창의성의 상한입니다.

```
규칙 1개: 표현 공간 = 100%
규칙 2개: 표현 공간 = 50% (각 규칙이 절반씩 잘라냄)
규칙 10개: 표현 공간 = 수 %
규칙 33개: 표현 공간 ≈ 0%
```

(실제로는 독립적이지 않아 이렇게 단순하진 않지만, 대략의 감)

33개 규칙을 전부 만족시키는 조합은 **이미 거의 단일 점**에 수렴합니다. 그 단일 점에서 "다양성" 을 만들려니 당연히 안 됨.

**해결책**: 규칙을 줄이고 표현 공간을 늘린다. 단, **결과물 구조** 는 엄격히 검증.

### 6-3. 표현 층 vs 측정 층 분리 원칙

```
            | 표현 층 (과정)       | 측정 층 (결과)
────────────┼──────────────────────┼──────────────────
원칙        | 자유 극대화           | 엄격 고정
어제        | DSL (제약된 자유)     | 33 validator (느슨한 측정)
오늘        | TSX (완전 자유)       | 3 AST validator (엄격한 측정)
효과        | 창의성 폭발           | 품질 고정
```

**잘못된 접근**: 표현 층을 제약하고 측정 층도 엄격히. → 아무것도 못 만듦.
**잘못된 접근**: 표현 층 자유롭고 측정 층도 느슨. → 품질 제어 불가.
**옳은 접근**: 표현 층 자유 + 측정 층 엄격. → 창의성 + 품질 공존. ✅

### 6-4. "자유와 규율의 공존" 의 실제

오늘의 워크플로우 체크리스트를 보면:
- [x] Claude 가 JSX 트리를 **자유롭게** 설계
- [x] `render-single-scene.sh` 로 PNG 생성
- [x] PNG 를 **이미지로 Read** (육안 검수)
- [x] `⓪-tsx1/2/3` AST validator 자동 실행
- [x] pre-commit hook 차단
- [x] 통과하면 commit

**자유 설계 + 엄격 검증 = 창의적이고 품질 고정된 결과**.

## Part 7. 반복 가능한 패턴

이 경험을 다른 문제에 일반화하면:

### 7-1. 체크리스트 — "규칙 추가" 전에 자문할 것

품질 문제를 겪을 때 아래 체크리스트를 먼저:

- [ ] **규칙이 이미 많은가?** (ex: 10개+)
- [ ] **규칙 추가 후 개선이 점점 작아지는가?** (diminishing returns)
- [ ] **결과물이 "평균에 수렴" 하는가?** (다양성 부족)
- [ ] **표현 층(어떻게 만드는가)을 제한하고 있는가?**
- [ ] **제한 제거 시 측정(결과 구조 검증) 가능한가?**

3개 이상 ✅ 면 → **규칙 추가가 아니라 표현 층 해방** 을 고려하세요.

### 7-2. 해방 시나리오

| 상황 | 해방 방법 |
|------|----------|
| 블록 기반 DSL → 단조로움 | 자유 코드 (TSX/Python/etc) 로 전환 |
| 템플릿 시스템 → 쌍둥이 | AI 에게 빈 종이 주고 설계 맡기기 |
| 폼 필드 제약 → 표현 한계 | 자유 텍스트 + 구조 추출 파이프라인 |
| 카테고리 태그 → 분류 오류 | 임베딩 + 의미 검색 |

### 7-3. 결과 측정 층 설계 원칙

자유를 주되 결과는 엄격히:

1. **구조 기반 검증** (AST, signature, schema)
2. **교차 검증** (여러 각도에서, ex: narration↔video↔subtitle 3각)
3. **자동 차단** (pre-commit hook, CI gate)
4. **육안 보조 검수** (PNG Read 같은)
5. **통계적 모니터링** (다양성 지표, 밀도 지표)

### 7-4. 안티패턴 — 해서는 안 되는 것

- ❌ **규칙을 줄이기만 하고 측정 안 함** → 품질 무제어
- ❌ **측정만 엄격하고 표현은 제한** → 아무것도 못 만듦
- ❌ **점진적 제약 완화** → 저항이 커서 과도기만 길어짐 (원칙 A 는 "단번에" 선언)
- ❌ **AI 능력 불신** → 최소 공통분모로 수렴

## Part 8. 다음 단계의 함의

### 8-1. 지속 가능한 개선 원칙

이 프로젝트가 앞으로 품질을 계속 올리려면:

1. **규칙 추가 자제** — 현재 AST validator 3종이면 충분. 추가하려면 기존을 하나 빼는 교환.
2. **Claude 모델 업그레이드 추적** — 1M context → 5M context 나오면 더 복잡한 씬 시퀀스 설계 가능.
3. **reference 확장** — 61장 → 200장 이상으로 늘리면 디자인 어휘 확장.
4. **측정 층 정교화** — PNG 기반 픽셀 검증을 mp4 기반 시퀀스 검증으로 확장 (감정 아크 일관성 등).

### 8-2. 주의할 점

- **1M context 가 없으면 이 해결책 재현 불가** — context window 가 디자인 능력의 전제.
- **강제력(pre-commit hook 등) 약화 시 즉시 후퇴** — 관성적으로 DSL 쓰는 습관 재발.
- **reference/SC *.png 의 Visual DNA 가 훼손되면 톤 일관성 무너짐** — 레퍼런스 관리 중요.

### 8-3. 아직 남은 문제

오늘 돌파는 **초반 12 씬** 에 한정. 전체 78 씬 중 22 씬은 여전히 DSL 상태 (`_dsl_rationale_review_required: true`). 다음 세션에서:

- 전체 78 씬 렌더 후 R11-audit (6 지표 전수 측정)
- TransitionSeries 말미 블랙 이슈 근본 해결 (Root.tsx `calculateDuration` 수정)
