## 합의 사항 (모든 리뷰어 동의)

### 1. 타입 안전성 — 최우선 해결 대상
- `StackNode`의 `data: Record<string, any>`가 60+개 노드의 props를 컴파일 타임에 검증하지 못함
- 생성기-렌더러-에디터 간 계약이 런타임으로 밀려 대규모 리팩토링 저해
- **대안**: 각 노드별 Zod 스키마 정의 → 컴파일 타임 검증 확보

### 2. 레지스트리 개선 — Build-time Codegen
- 플랫 `Record<string, FC>`는 메타데이터(props 스키마, 카테고리, 기본값, 버전)를 담지 못함
- 런타임 플러그인(import.meta.glob)은 Remotion의 webpack 기반 빌드에서 불안정
- **합의된 대안**: `nodes/[type]/meta.ts`(Zod 스키마 + 메타데이터) + 빌드 전 스크립트로 정적 `registry.ts` 자동 생성
  - 개발 경험은 auto-discovery처럼 편리
  - 런타임은 정적 import라 번들 예측 가능성, 타입 추적, 디버깅 모두 강함

### 3. layout_family vs stack_root 이중 아키텍처 — 설계 결함
- `SceneRenderer`가 선언형 렌더링(stack_root)과 switch 기반 라우팅(layout_family)을 동시에 책임
- 같은 Scene에서 어떤 필드가 진실인지 불명확 → 데이터 계약, 테스트, 편집기, 생성 파이프라인 이중화
- 단순 과도기가 아닌 "기술 부채의 함정" — DSL로의 완전한 전환을 방해
- **대안**: stack_root로 완전 통합 후 layout_family는 stack_root 생성의 힌트(메타데이터)로만 유지

### 4. Scene 타입 비대화 — God Object
- 타임라인, 카피, 배경, 샷플랜, 전환, stack tree가 단일 타입에 혼재
- Optional 필드 과다 → 상태 의미 불명확, 검증 포인트 분산
- **대안**: 관심사별 서브타입 분리 (TimelineInfo, VisualConfig, NarrationConfig 등)

### 5. 모션 시스템 경직성
- `computeMotionStyle()`의 대형 switch는 preset 추가는 쉬우나 조합성/테스트성 부족
- entrance/emphasis/exit가 독립 파이프라인이 아닌 문자열 분기에 묶임
- **대안**: 모션 프리셋을 독립 함수 객체로 분리, 조합 가능한 파이프라인 구조로 전환
